Problem Framing

Why Does IoT Exist?
The World Before the Network.

Before chips and protocols, look at the problem. Every great technology is born from a painful gap between what humans can observe and what they need to know — right now, from anywhere.

Imagine a world where every piece of information about the physical environment — the temperature of a warehouse, the pressure in a pipe, the heartbeat of a patient, the moisture level of a field — exists only locally, at the source, invisible to everyone not standing next to it. That was the world before IoT. And in many critical situations, information that arrives late is information that arrives too late.

🌾
Story 1 — The Farmer's Wasted Day
A farmer drives three hours to check an irrigation pump. The pump is fine. He drives three hours back. The following week, the same pump fails silently overnight and floods a field before dawn. Total loss: one season's crop. A $4 soil moisture sensor and a cellular module could have sent an SMS alert at the exact moment the anomaly began. The information existed — inside the machine. No one could hear it.
🏥
Story 2 — The Silent Ward
A hospital manages 40 patients per nurse at night. A post-operative patient's oxygen saturation drops below safe levels at 3 AM. The bedside monitor alarms locally — but the nurse is occupied two rooms away with another emergency. Six minutes pass before anyone reaches the bed. Continuous IoT monitoring with real-time push alerts changes the response window from minutes to seconds. That gap is the difference between recovery and permanent damage.
🏭
Story 3 — The Machine That Nobody Listened To
A manufacturing plant runs high-speed compressors 24/7. Vibration patterns shift subtly over 72 hours — a classic signature of bearing failure. Nobody notices. On a Thursday afternoon the machine seizes, halting an entire production line. Downtime cost: $40,000. A vibration sensor deployed on an edge device would have flagged the drift 48 hours earlier with 94% accuracy. The machine was trying to tell someone. No one was listening.

What Exactly Is IoT?

The term Internet of Things was coined in 1999 by Kevin Ashton while working at Procter & Gamble, in the context of RFID supply chain tracking. His insight: computers depended entirely on humans to enter information. IoT flips this — the devices themselves observe the world and generate data autonomously.

A clean one-line definition: "Giving everyday objects the ability to sense, communicate, and be controlled over a network."

Today, over 18 billion IoT devices are connected globally (IoT Analytics, 2024). By 2030, projections exceed 40 billion — more than four connected devices for every person on Earth. The IoT market is not a future trend. It is present infrastructure.

Core Concepts

How IoT Actually Works:
The 4-Layer Architecture

Every IoT system — from a $3 soil sensor to a million-dollar industrial controller — follows the same fundamental architecture. Build this mental model first. Everything else is detail.

Layer 1 — Perception (Sensors & Actuators)

Sensors translate physical phenomena into electrical signals. Actuators do the reverse — they turn signals into physical action. This layer is the IoT system's contact with the physical world.

🔍 Common Sensors
  • DHT11/DHT22 — Temperature & humidity (digital, 1-wire-like protocol)
  • HC-SR04 — Ultrasonic distance (TRIG pulse → ECHO duration, up to 4m)
  • MPU6050 — 3-axis accelerometer + 3-axis gyroscope (I²C, 0x68)
  • BMP180/BMP280 — Barometric pressure + temperature (I²C)
  • Soil Moisture — Resistive or capacitive; analog voltage output
  • PIR HC-SR501 — Passive infrared motion detector (digital output)
  • LDR — Light Dependent Resistor; read via voltage divider + ADC
⚡ Common Actuators
  • LED — Status indicator, PWM dimming, RGB color control
  • Servo Motor SG90 — Positional control 0°–180° via PWM pulse width
  • DC Motor + L298N — Variable speed and direction via H-bridge driver
  • Relay Module — Switch real-world AC loads safely from 3.3V logic
  • Buzzer — Audio alerts, tone generation via digital/PWM pin
  • OLED SSD1306 — 128×64px display over I²C; great for local data readout

Layer 2 — Processing (The Microcontroller)

The microcontroller (MCU) is the brain of every IoT node. It reads electrical signals from sensors, executes your firmware to make decisions, drives actuators, and manages communication. Unlike a general-purpose computer, an MCU runs a single program continuously — the firmware loop never stops as long as power is applied. This is intentional: reliability and determinism matter more than flexibility in embedded systems.

The MCU bridges the physical and digital worlds. On one side, raw voltages from sensors. On the other, structured data packets destined for a network. Everything in between — filtering noise, converting units, checking thresholds, triggering alerts — is the MCU's job.

Key MCU subsystems you will interact with as an IoT developer:

🧠 CPU Core
Executes your code instruction by instruction. On Arduino Uno: 8-bit AVR at 16 MHz (~16 million instructions/second). On ESP32: dual 32-bit Xtensa cores at 240 MHz. Clock speed determines how fast decisions are made — but most IoT tasks are I/O-bound, not CPU-bound.
💾 Memory Types
Flash — stores your program permanently (32 KB on Uno, 4 MB on ESP32). SRAM — working memory for variables at runtime (2 KB on Uno, 520 KB on ESP32). EEPROM — small non-volatile storage for settings that survive power cuts (1 KB on Uno).
⏱ Timers & Watchdog
Hardware timers generate precise time intervals independently of the CPU — used for PWM generation, baud rate clocking, and scheduling. The watchdog timer automatically resets the MCU if the firmware hangs — critical for unattended field deployments.
💡
Why do MCUs have so little RAM?
The Arduino Uno has 2 KB of SRAM. That sounds absurdly small — your browser tab uses thousands of times more. But an MCU reading a temperature sensor and sending MQTT packets needs to hold perhaps 50 bytes of variables at any time. The constraint forces lean, efficient code and dramatically reduces power consumption and cost. Constrained resources are a feature of embedded design, not a bug.

Layer 3 — Connectivity (The Network)

Layer 3 is where your device stops being isolated and becomes part of a system. Data leaves the MCU and travels — across a room over BLE, across a city over LoRa, or across the planet over the internet via Wi-Fi and MQTT. The choice of connectivity protocol here is the single most consequential design decision in most IoT projects, because it determines range, power budget, data rate, and infrastructure cost simultaneously.

These constraints form an inescapable triangle: you can optimize for long range, low power, or high data rate — but not all three at once. Every protocol is an engineered trade-off within this triangle.

Beyond choosing a radio protocol, Layer 3 also involves the application-layer messaging protocol that rides on top of the network. For IoT, this is almost always MQTT (for constrained devices over TCP) or HTTP/REST (for cloud integrations). We cover both in detail in the Protocols section.

A key concept at this layer is the difference between always-on and duty-cycled connectivity. A mains-powered device can hold a Wi-Fi connection permanently. A battery-powered field sensor must wake up, transmit in milliseconds, and return to deep sleep — otherwise battery life collapses from years to days.

📶 Always-On Connection
Device maintains a persistent network connection. Receives commands instantly. Can stream continuous data. Requires stable power supply. Best for: mains-powered devices, gateways, indoor monitors, devices that need to respond to remote commands in real time.
💤 Duty-Cycled / Deep Sleep
Device wakes every N minutes, transmits a sensor reading in <1 second, goes back to sleep. ESP32 deep sleep current: ~10 µA. A 3000 mAh battery can last 2–5 years this way. Best for: remote sensors, weather stations, agricultural monitors, anything battery-powered.

Layer 4 — Application (What Users See)

Layer 4 is where raw data becomes value. A temperature reading of 27.3 is just a number. A dashboard showing a 72-hour trend, firing a push notification when the reading exceeds 35°C, and letting a facilities manager remotely activate a cooling unit — that is value. Layer 4 is the reason any of the other three layers exist.

The application layer encompasses everything the end user interacts with: dashboards, mobile apps, email and SMS alert systems, automation rules (if soil moisture < 30% → trigger pump), and increasingly, AI/ML inference — running machine learning models on cloud or edge to extract patterns from sensor data that humans cannot see in raw numbers.

Key Vocabulary — Essential Terms

GPIO
General Purpose Input/Output. The physical pins you control via code. Each pin can be: digital input, digital output, analog input (via ADC), PWM output, or a special function (UART/SPI/I²C). Most confusion in beginner projects comes from wrong GPIO configuration.
PWM
Pulse-Width Modulation. Simulates analog output on digital pins by switching rapidly between ON and OFF. The ratio of ON time to total period is the duty cycle (0–100%). Used for LED dimming, motor speed, servo position.
ADC / DAC
ADC reads analog voltage and converts to a digital number (0–1023 on 10-bit Uno; 0–4095 on 12-bit ESP32). DAC outputs a precise analog voltage from a digital value. ESP32 has 2 DAC channels; Arduino Uno has none.
Firmware
Software permanently written to a device's flash memory. Runs directly on hardware with no OS in between (for MCUs). The firmware loop runs continuously — every millisecond, every hour, until power is removed.
Edge vs Cloud
Edge: Decision on or near the device. Fast, private, works offline. Cloud: Data sent to remote server. Scalable, powerful, requires connectivity. Real systems use both — edge for real-time response, cloud for historical analysis.
Interrupt
A hardware signal that pauses normal execution to handle an urgent event. Example: a button press triggers an interrupt immediately — no polling needed. Without interrupts, fast events would be missed while loop() is doing something else.

Hardware

Boards, Pins & Silicon:
The Physical Layer

Four boards dominate the beginner-to-intermediate IoT space. Each has a distinct personality. Knowing when to use which one is a fundamental IoT skill.

Arduino Uno R3 — The Learning Board

The Arduino Uno is the universal entry point to microcontroller programming. Based on the ATmega328P — an 8-bit AVR MCU at 16 MHz — it has no OS, no Wi-Fi, no Bluetooth. What it has is radical simplicity: write code, upload it, it runs. Forever. That's why it's taught first.

The Uno is the best board for learning fundamental embedded concepts precisely because it strips away abstraction. Every pin, every byte of memory, every cycle matters. Official Arduino Uno R3 Documentation

ESP32 (Espressif) — The IoT Workhorse

The ESP32 is the most important board you will learn. A dual-core 240 MHz Xtensa LX6 processor, 520 KB SRAM, built-in Wi-Fi 802.11 b/g/n, Bluetooth Classic, and BLE — all for $5–8. It can be programmed with Arduino IDE, MicroPython, or Espressif's native IDF. Nothing else at this price point comes close for IoT prototyping.

STM32 (STMicroelectronics) — The Professional's MCU

The STM32 family is where hobbyist meets professional. Built on the ARM Cortex-M architecture, STM32 devices power consumer electronics, medical devices, and industrial automation. The most accessible entry point: the STM32F103C8T6 "Blue Pill" and the official NUCLEO development boards.

STM32 is 32-bit (vs Arduino's 8-bit), runs at 72–480 MHz depending on variant, has hardware floating-point — essential for signal processing and control — and deterministic timing via hardware timers. Programmed with STM32CubeIDE, or the Arduino IDE with STM32 core. STM32 Family Overview

  • 64–128 KB Flash · 20 KB SRAM
  • 37 GPIO pins · 10 ADC channels (12-bit)
  • 3× USART · 2× SPI · 2× I²C · CAN bus
  • 4× 16-bit timers + 1× 32-bit general timer

When to Choose STM32
  • Precise timing — motor control, DAQ, PID loops
  • Hardware FPU — DSP, sensor fusion, Kalman filter
  • Building a product — not a prototype
  • Low power sleep — <1 µA in Stop mode
  • DMA — high-speed data transfer without CPU
  • Multiple buses simultaneously — SPI + I²C + UART + PWM

Raspberry Pi — The Linux Single-Board Computer

The Raspberry Pi is not a microcontroller — it's a full Linux single-board computer. It runs Raspberry Pi OS (Debian-based), has a desktop environment, runs Python/Node.js/C++, and connects to monitors, keyboards, and the internet like any laptop.

In IoT, the Pi plays specific roles: IoT gateway (aggregates data from multiple MCU nodes), local server (runs MQTT broker, InfluxDB, Grafana), vision system (camera + OpenCV), or edge AI (TensorFlow Lite). It should not be the first choice for simple sensor reading — that's what MCUs are for. Raspberry Pi Official Documentation

⚠️
Critical: Raspberry Pi GPIO is 3.3V Only
Pi GPIO pins are 3.3V logic and cannot tolerate 5V signals. Applying 5V will permanently damage the GPIO. Max 16 mA per pin, 50 mA total across all GPIO. No built-in ADC — use an external MCP3008 via SPI. For real-time GPIO control, Pi is unreliable due to Linux OS scheduling — use an MCU for time-critical tasks.

int ledPin = 13;   // Global variables: accessible in both functions
int counter = 0;

// ── setup() ─────────────────────────────────────────────────────────
// Runs ONCE when the board powers on or resets.
// Use for: pinMode(), Serial.begin(), library init, one-time config.
void setup() {
  Serial.begin(115200);       // Open serial communication
  pinMode(ledPin, OUTPUT);    // Configure pin 13 as output
  Serial.println("Board started!");
}

// ── loop() ──────────────────────────────────────────────────────────
// Runs FOREVER after setup() completes. Never stops.
// This is where your program lives.
void loop() {
  digitalWrite(ledPin, HIGH);  // LED ON
  delay(1000);                  // Wait 1 second
  digitalWrite(ledPin, LOW);   // LED OFF
  delay(1000);
  counter++;
  Serial.print("Blink #");
  Serial.println(counter);
}

Protocols

How Devices Talk:
Hardware & Network Protocols

Two layers of communication: hardware protocols (chip-to-chip on a board) and network protocols (device-to-cloud across the internet). Both are essential.

Hardware Communication Protocols

UART
Universal Asynchronous Receiver/Transmitter
"The old-reliable — simple, universal, no clock needed"

UART uses just two wires: TX (Transmit) and RX (Receive). It's asynchronous — no shared clock. Both devices must pre-agree on a baud rate (bits/second). Common: 9600, 115200, 921600 bps. Data is sent one bit at a time, framed by start/stop bits.

On ESP32, UART0 is the USB programming port; UART1 and UART2 are free for GPS modules, GSM modems, Bluetooth HC-05 serial modules, and inter-MCU communication. ESP32 UART Guide

2 wires: TX + RXFull duplexNo master/slavePoint-to-point onlyUp to ~5 MbpsNo addressing

Use for: GPS module, GSM modem, Bluetooth HC-05/06, Serial Monitor debug, MCU-to-MCU link.
Watch out: Baud rate mismatch (garbled data), 5V/3.3V level mismatch (use a level shifter!).

SPI
Serial Peripheral Interface
"The speed demon — fastest common hardware bus"

SPI is synchronous and full-duplex. It uses a shared clock (SCK) controlled by the master, MOSI (data to slave), MISO (data from slave), and one SS/CS (Slave Select) line per slave — pulled LOW to activate. Multiple slaves can share MOSI/MISO/SCK but each needs its own SS line.

It's the fastest hardware protocol — up to 80 MHz on ESP32 — ideal for SD cards, TFT displays, LoRa modules (SX127x), and high-speed ADCs.

4 wires: MOSI+MISO+SCK+SSSynchronousFull duplexMaster-slaveUp to 80 MHz1 SS per slave
SPI timing — CS goes LOW, master clocks 8 bits synchronously. MOSI = master→slave, MISO = slave→master simultaneously.

I²C
Inter-Integrated Circuit
"The bus — many sensors, only two wires"

I²C uses only two wires: SDA (data) and SCL (clock). Up to 127 devices share the same two wires, each with a unique 7-bit address. It's synchronous and half-duplex. Typical speeds: 100 kHz (Standard), 400 kHz (Fast), 3.4 MHz (High-Speed).

Most popular sensors use I²C: MPU6050 (IMU, 0x68), BMP280 (pressure, 0x76/0x77), SSD1306 OLED (0x3C/0x3D), ADS1115 (ADC, 0x48). Always pull SDA and SCL up to VCC with 4.7kΩ resistors — I²C is open-drain and won't work without them.

Network Communication Protocols

Wi-Fi
IEEE 802.11 b/g/n/ac
"The highway — fast, familiar, power-hungry"

Wi-Fi delivers high bandwidth (up to 150 Mbps for 802.11n) at 2.4 GHz or 5 GHz, with 30–100m indoor range. The ESP32 connects to your router like any laptop — enabling HTTP, MQTT, WebSockets, and more with minimal setup.

Power concern: ESP32 with Wi-Fi TX draws ~240 mA. On a 3000mAh Li-ion, that's ~12 active hours. For battery-powered field sensors, use BLE or LoRa. For mains-powered devices, Wi-Fi is the right default choice.

Range: ~50m indoorsSpeed: up to 150 MbpsPower: ~240 mA TX2.4/5 GHzFull TCP/IP stack
BLE
Bluetooth Low Energy (BT 4.0+)
"The whisper — battery-sipping, short range"

BLE was designed for coin-cell-powered sensors transmitting small payloads infrequently. A BLE sensor can run 1–2 years on a CR2032. It operates at 2.4 GHz with 1–3 Mbps data rate and 10–100m range.

BLE uses the GATT model: devices expose Services (logical groups) containing Characteristics (data values). Your phone reads or writes characteristics from a peripheral. ESP32 can act as BLE server (peripheral) or client (central). Use the free nRF Connect app (Android/iOS) to debug BLE devices. ESP32 BLE Guide

Range: 10–100mSpeed: 1 MbpsPower: ~7–15 mA TX2.4 GHz ISMGATT protocol

LoRa
Long Range (Chirp Spread Spectrum)
"The long-distance runner — km range on a coin cell"

LoRa uses proprietary chirp-spread-spectrum modulation (Semtech). It trades throughput (0.3–50 kbps) for extraordinary range — 2–15 km rural, 1–3 km urban — while drawing only 15–30 mA transmitting. A coin cell can power a LoRa node for over a year with infrequent transmissions.

LoRaWAN adds a network protocol layer on top: devices → gateways → network server → application server. A single LoRaWAN gateway can cover an entire city. Common hardware: SX1278 (RA-02 module, 433 MHz for Nepal/Asia). arduino-LoRa library · The Things Network (TTN)

Range: 2–15 kmSpeed: 0.3–50 kbpsPower: ~15 mA TXSub-GHz: 433/868/915 MHzSPI-connected module

MQTT
Message Queuing Telemetry Transport
"The postal service of IoT — publisher/subscriber model"

MQTT runs over TCP/IP and uses a broker as a central relay. Devices publish messages to topics (like home/bedroom/temp); devices that have subscribed to that topic receive them instantly. The publisher and subscriber never talk directly — the broker is the post office.

Key features: QoS levels (0=at most once, 1=at least once, 2=exactly once), retained messages (broker stores last value for new subscribers), Last Will & Testament (offline alert), and an incredibly lightweight 2-byte fixed header. mqtt.org · Free public broker: HiveMQ

Sound travels at ~343 m/s. The sensor emits an ultrasonic burst at 40 kHz via the TRIG pin, then measures how long the ECHO pin stays HIGH. Half that time × speed of sound = distance to object.

GND→ GND

GND→ GND

↓ Download
// HC-SR04 Ultrasonic Distance — works on Uno and ESP32
#define TRIG_PIN 9
#define ECHO_PIN 10

void setup() {
  Serial.begin(115200);
  pinMode(TRIG_PIN, OUTPUT);
  pinMode(ECHO_PIN, INPUT);
}

float getDistanceCm() {
  // Send 10µs trigger pulse
  digitalWrite(TRIG_PIN, LOW);
  delayMicroseconds(2);
  digitalWrite(TRIG_PIN, HIGH);
  delayMicroseconds(10);
  digitalWrite(TRIG_PIN, LOW);

  // Measure echo pulse (30ms timeout = ~5m max range)
  long dur = pulseIn(ECHO_PIN, HIGH, 30000);
  if (dur == 0) return -1;        // No echo = out of range

  // Distance = (343 m/s × time) / 2   →   0.0343 cm/µs
  return (dur * 0.0343f) / 2.0f;
}

void loop() {
  float d = getDistanceCm();
  if      (d < 0)   Serial.println("Out of range!");
  else if (d < 10)  Serial.printf("%.1f cm  ⚠️  VERY CLOSE!\n", d);
  else if (d < 30)  Serial.printf("%.1f cm  NEAR\n", d);
  else               Serial.printf("%.1f cm\n", d);
  delay(200);  // Min 60 ms between readings
}

#include <MPU6050.h>

MPU6050 mpu;

  Wire.begin();

  int16_t ax, ay, az, gx, gy, gz;
  mpu.getMotion6(&ax, &ay, &az, &gx, &gy, &gz);

  // Accel: divide by 16384 for ±2g range  →  result in g

  html += "<div class='card'><div class='val'>" + String(g_dist,1) + "cm</div><div class='lbl'>📡 Distance</div></div>";
  html += "<div class='card'><div class='val'>" + String(g_moisture) + "%</div><div class='lbl'>🌱 Soil Moisture</div></div>";

  html += "</div><div class='ts'>Uptime: " + String(millis()/1000) +
          "s  ·  Auto-refreshes every 3s</div></body></html>";

  server.send(200, "text/html", html);
}

void handleAPI() {
  readSensors();
  char buf[128];
  snprintf(buf, sizeof(buf),
    "{\"temp\":%.1f,\"hum\":%.1f,\"dist\":%.1f,\"moisture\":%d,\"uptime\":%lu}",
    g_temp, g_hum, g_dist, g_moisture, millis()/1000);
  server.send(200, "application/json", buf);
}

void setup() {
  Serial.begin(115200);
  pinMode(TRIG_PIN, OUTPUT);
  pinMode(ECHO_PIN, INPUT);
  dht.begin();

  WiFi.begin(ssid, password);
  Serial.print("Connecting to WiFi");
  while (WiFi.status() != WL_CONNECTED) { delay(500); Serial.print("."); }

  Serial.printf("\n\n✅ Connected! Open browser → http://%s\n",
               WiFi.localIP().toString().c_str());

  server.on("/",    handleRoot);  // HTML dashboard
  server.on("/api", handleAPI);   // JSON API for integration
  server.begin();
  Serial.println("Web server listening on port 80.");
}

void loop() {
  server.handleClient();  // Process incoming HTTP requests
}

Security & Ethics

The Dark Side of Connected Devices:
Group Workshop Activity

This section is a structured group exercise. Analyze real-world attack scenarios, debate privacy implications, and leave with concrete security habits you can implement from day one as an IoT developer.

⚠️
Real Incident — The Mirai Botnet (October 2016)
Malware called Mirai infected over 300,000 IoT devices — security cameras, DVRs, home routers — by scanning the internet for devices still using factory default passwords. These compromised devices were weaponized to launch one of the largest DDoS attacks in history, taking down Dyn DNS and making Twitter, Netflix, Reddit, CNN, and GitHub inaccessible for hours across the US East Coast. The entire attack relied on a hardcoded list of just 61 default username/password pairs. The fix was trivially simple: change the default password. Cloudflare — Famous DDoS Attacks · Wired — The Mirai Story

Group Activity — 4 Scenarios, 4 Groups

Split into 4 groups. Analyze your scenario for 8 minutes — identify the attack vector, map the impact, and propose mitigations. Present in 3 minutes. The facilitator synthesizes the common thread across all four.

G1
Group 1 — The Smart Home Breach
Topic: Hardcoded credentials & factory defaults
Scenario: A company ships 200,000 Wi-Fi doorbell cameras. Every unit has the same hardcoded username admin and password admin123. A researcher buys one, reads the firmware from the flash chip, finds the credentials in plaintext. Posts them publicly. Within 48 hours, an automated scanner has logged into every device on the internet.

Discuss:
1. Beyond privacy invasion — what else can an attacker do with access to 200,000 cameras?
2. What three engineering decisions at the design stage would have prevented this entirely?
3. Who bears legal responsibility — the manufacturer, the retailer, or the consumer?
4. As a buyer: what questions would you ask before purchasing any connected device?
G2
Group 2 — The Unencrypted Hospital
Topic: Plaintext data transmission in critical infrastructure
Scenario: A hospital deploys 50 IoT patient monitors. They transmit vital signs — heart rate, blood pressure, SpO₂ — over unencrypted HTTP to the nurses' central dashboard. A researcher sitting on the hospital guest Wi-Fi runs Wireshark and captures complete patient records in plaintext — including names matched to room numbers.

Discuss:
1. Who might be motivated to intercept this data — and for what purpose?
2. Beyond HTTPS, list three additional security layers that should exist.
3. What regulations apply here? Research: HIPAA, ISO 27001, Nepal's Electronic Transactions Act.
4. More dangerous: an attacker who reads the data, or one who modifies the readings?
G3
Group 3 — The Data-Hungry Fridge
Topic: Data ownership, consent, and re-identification
Scenario: A smart refrigerator startup collects usage data: door open/close times, temperature settings, and camera snapshots when items run low. Their ToS (page 47 of 80) states they may share "aggregated, anonymized data with partners." Insurers purchase this data and use it as a lifestyle indicator for setting health premiums.

Discuss:
1. Is "aggregated and anonymized" truly safe from re-identification? Research: data re-identification attacks.
2. What data rights should users have? Research: GDPR's right to erasure.
3. Is this legal? Is it ethical? Are legal and ethical the same thing here?
4. If you were building this product: what would you collect, what would you refuse to collect, and why?
G4
Group 4 — The Unupdatable Sensor
Topic: No OTA update mechanism & end-of-life devices
Scenario: 100,000 water-quality IoT sensors are deployed across a country. Three years later, a critical vulnerability is found in the TLS library they use — an attacker can intercept and silently modify sensor readings. The devices have no OTA update mechanism. Physical access is required to patch each unit. 60% are in remote mountain regions.

Discuss:
1. What happens if drinking water quality readings can be falsified by an attacker?
2. Why do manufacturers skip OTA update capability? (Hint: cost, complexity, liability, battery life.)
3. Design a minimal OTA system for an ESP32. Research: ESP32 ArduinoOTA guide.
4. Who should regulate this gap — government mandates, industry bodies, or market forces?

The Three Deadly Sins of IoT Security

🔑
Sin 1 — Hardcoded Credentials
Never ship a device with a fixed username/password. Generate unique per-device credentials at manufacturing, or force a password change on first boot. Treat any factory credential as a known public exploit — because it will eventually become one.
🔓
Sin 2 — Plaintext Transmission
HTTP → HTTPS. MQTT → MQTT over TLS (port 8883). Any data over a network must be encrypted. Use TLS 1.2+ minimum. Free certificates via Let's Encrypt. If it's too inconvenient to encrypt, the system isn't ready to deploy.
🔄
Sin 3 — No Update Path
All software has bugs. All bugs may become vulnerabilities. Build OTA update capability from day one — it's far cheaper than a hardware recall. ESP32's ArduinoOTA library adds this in under 20 lines. There is no excuse for skipping it.

Developer Security Checklist — Implement From Day 1

  • Change all default passwords on every device, every router, every MQTT broker you deploy
  • Use TLS/HTTPS everywhere — MQTT on port 8883, HTTP API calls over HTTPS, never plain HTTP
  • Never expose devices directly to the internet — use VPN, reverse proxy (nginx), or a managed cloud broker
  • Store credentials in config, not in code — never commit passwords or API keys to GitHub. Use .env files or device-specific config storage
  • Validate all incoming data — buffer overflows, integer overflows, and injection attacks hit firmware too, not just web servers
  • Implement OTA firmware updates — plan for it before deployment, not after a vulnerability is found
  • Collect minimum data — every data point you store is a liability if breached. Only gather what you demonstrably use
  • Know the law that applies to you — Nepal's Electronic Transactions Act 2063, GDPR if touching EU user data, HIPAA for health data
  • Disable unused services — if your device doesn't need Telnet, SSH, or FTP, don't run them. Every open port is an attack surface
  • Plan for end-of-life — what happens to user data when your product is discontinued? State this policy before launch
